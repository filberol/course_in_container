# Руководство по созданию презентаций курса

Курс: **«Программная инфраструктура и управление разработкой»** (Университет ИТМО).
Взгляд курса — **системный аналитик инфраструктуры**; сквозная аналитическая рамка каждой лекции:
**проблема → модель → границы → критерии выбора → режимы отказа → свидетельства**.
Сквозной пример курса — многосервисное приложение voting-app. Курсы-предшественники: ОПИ (Git,
профилирование) и «Технологии виртуализации» (гипервизоры, ВМ) — их темы не пересказываются,
а обозначаются коротким «слайдом-мостом».

## Входные материалы для каждой лекции

1. Конспект лекции: `../Лекции/Лекция N. <название>.md` — покрыть **каждую** тему из него.
2. План слайдов: `../Лекции/_Список слайдов.md` (раздел своей лекции) — ориентир по составу
   и порядку; каждый пункт плана разворачивается в 1–3 слайда. Углублять и добавлять слайды можно
   и нужно, выбрасывать темы нельзя.

## Формат файла

Один файл `slides/lectures/NN.md` (NN — номер лекции с ведущим нулём). Слайды разделяются `---`
на отдельной строке. Шапка файла:

```yaml
---
theme: ../theme
title: Лекция N. <Точное название лекции из файла конспекта>
info: |
  Курс «Программная инфраструктура и управление разработкой», Университет ИТМО
lang: ru
mdc: true
drawings:
  persist: false
layout: cover
---
```

## Обязательная структура презентации

1. **Обложка** (`layout: cover`, задан в шапке): `<div class="cover-kicker">Лекция N</div>`,
   затем `# Название`, затем один абзац-подзаголовок (интрига лекции, 5–10 слов).
2. **Маршрут лекции**: слайд со списком 4–6 смысловых блоков.
3. **Проблема**: с чего начинается лекция — какую боль решаем.
4. **Содержательные блоки**, разделённые слайдами `layout: section` с
   `<div class="section-no">01</div>` и заголовком блока.
5. **Критерии выбора** — таблица решений (фирменный приём курса).
6. **Режимы отказа** — карточки `itmo-card-warn`.
7. **Свидетельства** — как проверить руками (`docker inspect`, `kubectl describe`, логи…),
   мост к лабораторной, если он есть в конспекте.
8. **Итоги** (`layout: center`): 3–5 выводов и «Дальше: …» — мост к следующей лекции.
   Последней строкой — опорная литература полными названиями, например:
   «Опорная литература: Дж. Ким и соавт. „Руководство по DevOps“».

Объём: **25–40 слайдов**. Один слайд — одна мысль.

## Визуальность — главное требование

Каждый содержательный слайд должен нести визуальный элемент; слайды из одних булетов запрещены
(булеты допустимы как дополнение к диаграмме/таблице или на слайдах «Маршрут» и «Итоги»).

Приоритет средств:

1. **Mermaid-диаграммы** — основной инструмент. Подходящие типы:
   - `flowchart LR/TB` — потоки, архитектуры, конвейеры;
   - `sequenceDiagram` — взаимодействия компонентов;
   - `stateDiagram-v2` — жизненные циклы (Pod, контейнер, релиз);
   - `classDiagram` — структуры объектов (ресурсы Kubernetes, спецификации);
   - `erDiagram` — связи сущностей;
   - `gitGraph` — стратегии ветвления (лекция 9);
   - `pie`, `timeline`, `mindmap` — умеренно, где уместно.
   Не использовать: sankey, xychart, quadrantChart, journey (нестабильны в рендере).
2. **Таблицы** — сравнения и критерии выбора (до 6 строк, до 4 колонок).
3. **Карточки** в сетке UnoCSS (`<div class="grid grid-cols-2 gap-3">…`):
   `itmo-card` (нейтральная), `itmo-card-accent` (ключевой вывод),
   `itmo-card-warn` (риск/режим отказа), `itmo-card-note` (практический совет).
4. **Код** — только когда без него нельзя (Dockerfile, YAML): ≤ 12 строк, с указанием языка.

### Правила для Mermaid

- Размер: `\`\`\`mermaid {scale: 0.7}` — подбирать 0.5–0.9 так, чтобы диаграмма помещалась
  (больше узлов → меньше scale).
- Не более ~10 узлов на диаграмму; подписи короткие, по-русски (термины — как в конспекте).
- Подписи со скобками/спецсимволами брать в кавычки: `A["kubelet (агент узла)"]`.
- Перенос строки внутри подписи узла — только `<br/>`; последовательность `\n` НЕ работает
  (отрисуется буквально как «\n»).
- Не смешивать синтаксисы рёбер: либо `A -->|"подпись"| B`, либо `A -- подпись --> B`,
  но не оба на одном ребре.
- Не оставлять в flowchart два и более subgraph без единого ребра между ними —
  диаграмма может отрисоваться пустой; связывать подграфы ребром или заменять карточками.
- Цвета не задавать — тема оформляет диаграммы сама.

## Заметки докладчика — полный текст выступления

В конце **каждого** слайда — HTML-комментарий с речью лектора:

```
<!--
Полный связный текст того, что лектор говорит на этом слайде: 60–140 слов,
разговорный академический стиль, «мы» вместо «вы должны», проговаривание каждого
элемента диаграммы, связки с предыдущим и следующим слайдом.
-->
```

- Речь должна быть самодостаточной: по заметкам можно прочитать лекцию, не готовясь.
- Внутри комментария нельзя использовать последовательность из двух дефисов `--`
  (ломает HTML-комментарий); тире писать символом «—».
- Книги упоминать естественно: «в „Руководстве по DevOps“ это называется…», без кодов
  вида [DOH] — они в курсе отменены.

## Стиль и язык

- Всё по-русски; устоявшиеся англоязычные термины не переводить (Pod, Deployment, trunk-based,
  rolling update, pull request).
- **Запрещены заголовки-конструкции «X как Y»** («Git как контур доставки») — это клише;
  формулировать иначе: «Git в контуре доставки», «Наблюдаемость: три сигнала».
- Названия лекций уже отредактированы — брать точно из файлов конспектов.
- Без эмодзи. Без восклицательных предложений на слайдах.
- На слайде ≤ 9 строк текста (не считая диаграмм/таблиц); длинные мысли — в заметки докладчика.

## Технические ловушки Slidev

- Символ `<` в тексте слайда воспринимается как HTML — писать `&lt;` или переформулировать.
- Двойные фигурные скобки в тексте интерполируются Vue — избегать; в коде Helm-шаблонов
  оборачивать блок в ` ```yaml ` (в fenced-блоках безопасно) или не использовать.
- `---` внутри слайда (например, в YAML-примере) допустим только в fenced-блоке.
- Именованный слот правой колонки: строка `::right::` (только в `layout: two-cols`).

## Самопроверка (обязательно)

Из каталога `slides/` (Node 24: `export PATH="$HOME/.nvm/versions/node/v24.15.0/bin:$PATH"`):

```bash
npx slidev export lectures/NN.md --output dist/lecture-NN.pdf --timeout 120000
```

1. Экспорт должен завершиться без ошибок.
2. Открыть PDF (инструментом Read, партиями по ≤ 20 страниц) и просмотреть **все** страницы:
   ни одна диаграмма/таблица не обрезана, текст не вылезает за нижний край, mermaid
   отрисовался (нет слайдов с сырым текстом диаграммы или сообщением об ошибке).
3. Починить найденное (уменьшить scale, разбить слайд на два, сократить текст) и повторить
   экспорт до чистого результата.
