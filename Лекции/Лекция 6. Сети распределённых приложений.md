# Лекция 6. Сети распределённых приложений

## О чём лекция
Лекция разбирает, как контейнеры и сервисы находят и вызывают друг друга, начиная с примитивов сети Linux и заканчивая балансировкой и service mesh.

## Проблема
В распределённой системе адреса сервисов меняются при пересоздании, а вызовы между ними должны оставаться устойчивыми. Нужен уровень, который скрывает подвижность адресов.

## Основы сети Linux
Сеть контейнера строится из примитивов ядра. network namespace даёт отдельный сетевой стек, veth-пара это виртуальный кабель между namespace контейнера и хостом, Linux bridge соединяет несколько veth в общий сегмент. Контейнер получает интерфейс, а bridge выполняет роль коммутатора.

## netfilter и iptables
Ядро фильтрует и преобразует трафик через netfilter. Правила iptables организованы в цепочки и выполняют трансляцию адресов: DNAT подменяет адрес назначения при пробросе порта, SNAT подменяет источник при выходе наружу. На этих правилах строятся публикация сервисов и маршрутизация.

## Docker-сети
Docker предлагает несколько сетевых драйверов. bridge это сеть по умолчанию на одном хосте, host отдаёт контейнеру сеть хоста без изоляции, none оставляет контейнер без сети, overlay строит сеть поверх нескольких хостов, macvlan выдаёт контейнеру собственный MAC в физической сети. Драйвер выбирают под сценарий.

## Порты и трансляция адресов
На хосте Docker создаёт мост docker0, к нему подключаются контейнеры. Проброс порта флагом -p настраивает DNAT с хоста в контейнер. Контейнеры одной сети общаются по внутренним адресам напрямую.

## DNS и обнаружение сервисов
Обнаружение сервисов это уровень косвенности между именем сервиса и его текущим адресом. Клиент обращается по имени, а система резолвит его в актуальный адрес, поэтому смена адреса реплики не ломает вызовы.

## docker-compose
compose описывает многосервисное приложение и запускает его на одном хосте. Он создаёт общую сеть, поднимает сервисы, связывает их по именам и задаёт переменные окружения. Это оркестрация уровня одного хоста.

## Балансировка нагрузки
Балансировщик распределяет запросы между репликами. L4 работает на уровне TCP или UDP и не видит содержимого. L7 понимает HTTP и маршрутизирует по пути, заголовку, хосту. Маршрутизация с учётом здоровья реплики выводит из ротации нездоровые экземпляры.

## Ingress и egress
Ingress это управляемый вход в систему снаружи, обычно через обратный прокси, который терминирует TLS и маршрутизирует запросы. Egress это контролируемый выход наружу с ограничением, куда сервисам можно обращаться.

## Service mesh
Service mesh выносит сетевую логику, то есть повторы, взаимный TLS, трассировку, из кода приложения в сопроводительный прокси sidecar. Приложение не знает о деталях сети, а политику задают централизованно.

## Критерии
На одном хосте достаточно compose и docker-сетей. С ростом числа узлов и требований к обнаружению сервисов и сетевой политике переходят к кластерной сети.

## Режимы отказа
Сбои DNS, таймауты, ретрай-штормы, когда лавина повторов усиливает перегрузку, разбиение сети, когда части системы теряют связь.

## Свидетельства
docker network inspect показывает состав сети, трассировка пути запроса и анализ задержек выявляют узкие места. Навык закрепляется в Лаб 1.

## Итог
Сеть это контракт взаимодействия сервисов. Понимание примитивов Linux нужно, чтобы диагностировать проблемы на любом уровне.
