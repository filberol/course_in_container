# Лекция 9. Git в контуре доставки

## О чём лекция
Лекция рассматривает Git не как систему хранения версий, а как модель доставки. Устройство Git считается пройденным в курсе ОПИ.

## Мост из ОПИ
Внутреннее устройство Git, коммиты, ветки, слияние и перебазирование, разрешение конфликтов изучены в курсе «Основы программной инженерии». Здесь эти знания входные, лекция посвящена влиянию работы с Git на доставку.

## Всё под контролем версий
В системе контроля версий держат не только код, но и конфигурацию, инфраструктуру, схему базы и сценарии сборки. Это делает состояние системы воспроизводимым и позволяет восстановить любую версию окружения.

## Малые партии
Чем меньше изменение, тем ниже риск и быстрее обратная связь. Крупные редкие слияния копят конфликты и усложняют поиск причины сбоя. Малые частые изменения интегрируются проще.

## Trunk-based development
В trunk-based разработке работают короткими ветками и часто вливают их в основную ветку. Основная ветка всегда пригодна к релизу, а незавершённую функциональность прячут за feature flags.

## Feature branching и release trains
Долгие ветки функциональностей и релизные поезда оправданы при строгих циклах релиза или регуляторных требованиях. Цена это отложенная интеграция и рост конфликтов.

## Сравнение стратегий
Выбор стратегии ветвления это выбор модели релиза. Trunk-based ускоряет поток и требует зрелой автоматизации. Ветвление даёт изоляцию ценой скорости интеграции.

## Pull Request как точка автоматизации
Pull Request это точка, где запускаются проверки: сборка, тесты, линтеры, сканеры. Обязательные статусы не дают влить изменение, пока проверки не пройдены. Так ревью и контроль качества встроены в процесс.

## Monorepo и polyrepo
В monorepo весь код в одном репозитории, что упрощает общие изменения и единые зависимости ценой масштаба сборки. В polyrepo сервисы разнесены, границы владения чётче, но сквозные изменения сложнее.

## GitOps
GitOps делает Git источником истины о желаемом состоянии инфраструктуры. Изменение вносят коммитом, а агент синхронизирует систему с репозиторием. История Git становится журналом изменений инфраструктуры.

## Конвенции
Conventional Commits задают формат сообщения коммита, semver задаёт правила версий. По ним автоматизация определяет тип изменения и вычисляет следующую версию.

## Режимы отказа
Расхождение долгих веток, накопление конфликтов при слиянии, нестабильная основная ветка, блокирующая всю команду.

## Свидетельства
Анализ истории, частота интеграции, время жизни веток показывают здоровье процесса доставки. Навык закрепляется в Лаб 1.

## Итог
Стратегия ветвления это стратегия релиза. Git организует поток изменений от разработчика до продакшена.

Опорная литература: К. Уилсон «Грокаем Continuous Delivery», Дж. Ким и соавт. «Руководство по DevOps».
