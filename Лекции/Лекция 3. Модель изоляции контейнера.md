# Лекция 3. Модель изоляции контейнера

## О чём лекция
Лекция разбирает, из чего складывается изоляция контейнера в Linux и какие гарантии она даёт. Контейнер это обычный процесс с ограничениями, наложенными ядром.

## Проблема
Нужна лёгкая изоляция процесса без веса полной виртуальной машины. Контейнер должен стартовать за доли секунды, потреблять мало ресурсов и при этом отделять приложение от соседей.

## Изоляция ядром против изоляции гипервизором
Виртуальная машина изолируется на аппаратной границе и несёт своё ядро. Контейнеры делят ядро хоста, поэтому изоляция слабее, а плотность и скорость выше. Общее ядро это ключевой компромисс контейнерной модели.

## Архитектура Docker
Запуск контейнера проходит через несколько компонентов [DBS]. Клиент docker отправляет команду демону dockerd, тот обращается к containerd, containerd через shim запускает runc, runc создаёт процесс с нужными namespaces и cgroups и завершает работу, а shim остаётся родителем процесса. Такое разделение позволяет перезапускать демон, не убивая контейнеры.

## Namespaces
Namespaces изолируют то, что процесс видит [DBS]. pid отделяет дерево процессов, net даёт свой сетевой стек и интерфейсы, mnt изолирует точки монтирования, uts имя хоста, ipc средства межпроцессного взаимодействия, user отображает идентификаторы пользователей, cgroup скрывает иерархию контрольных групп. Набор namespaces определяет, что контейнер считает своим окружением.

## Cgroups
Control groups ограничивают и учитывают ресурсы [DBS]. Подсистема cpu задаёт долю процессора, memory задаёт лимит памяти и порог OOM, blkio ограничивает дисковый ввод-вывод, pids ограничивает число процессов. Версия v2 объединяет подсистемы в единую иерархию. Cgroups превращают ресурсы в управляемую границу и ограничивают радиус поражения при сбое соседа.

## Привилегии
Linux capabilities дробят всевластие root на отдельные права, например CAP_NET_BIND_SERVICE, которые можно выдавать и снимать [DBS]. Seccomp ограничивает набор доступных системных вызовов. Модули безопасности AppArmor и SELinux задают политику доступа к файлам и ресурсам. Вместе они реализуют принцип наименьших привилегий.

## Файловая система
Корневая ФС контейнера собирается из слоёв только для чтения и одного слоя записи сверху. Ядро переключает процесс в этот корень через pivot_root. Слой записи существует, пока живёт контейнер.

## Спецификации OCI
Open Container Initiative задаёт два стандарта. image-spec описывает формат образа, runtime-spec описывает запуск контейнера. Благодаря им образы и рантаймы разных производителей совместимы, а runc и containerd это эталонные реализации.

## Границы изоляции
Контейнер не изолирует ядро: уязвимость ядра общая для всех контейнеров хоста. Привилегированный контейнер и лишние capabilities открывают путь к побегу. Доверие к образу это часть границы безопасности, потому что чужой слой может нести уязвимость.

## Критерии выбора
Контейнера достаточно для большинства сервисов с общим доверенным окружением. Для многопользовательской изоляции или запуска недоверенного кода применяют лёгкие виртуальные машины и песочницы (gVisor, Kata), которые возвращают аппаратную границу ценой части плотности.

## Режимы отказа
Типичные отказы: OOM-kill при превышении лимита памяти, троттлинг процессора, запуск в privileged без необходимости, эксплуатация уязвимости общего ядра.

## Свидетельства
Границы проверяются наблюдением. docker inspect показывает конфигурацию, каталог /proc и файлы cgroups раскрывают лимиты, ps aux сопоставляет процессы хоста и контейнера. Навык закрепляется в Лаб 1.

## Итог
Изоляция контейнера это сумма namespaces, cgroups, ограничений привилегий и слоистой файловой системы. Граница изоляции совпадает с границей доверия.
